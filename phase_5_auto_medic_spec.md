# Phase 5: Auto-Medic Specification
**Contract-Driven Zero-Downtime Remediation Bot**

## 1. Executive Summary and Current State
The DataOps CI/CD Gatekeeper has successfully completed Phases 1 through 4. The core orchestration modules—`main.py`, `parser.py`, and `api_client.py`—are fully validated. They accurately extract modified tables, query OpenMetadata for downstream lineage, and successfully block Pull Requests (exit code `100`) if a Tier-1 asset falls within the blast radius.

> **CRITICAL DIRECTIVE:** The foundational files (`main.py`, `parser.py`, and `api_client.py`) are strictly locked. They must not be modified under any circumstances.

Phase 5 represents a strategic pivot from reactive governance to proactive remediation. We are introducing a "Contract-Driven Zero-Downtime Remediation Bot" powered by a local LLM, operating entirely within a new isolated script: `remediation_agent.py`.

## 2. Architecture and Data Flow
The remediation bot operates as a secondary, decoupled step in the CI/CD pipeline, executing only after the primary blast radius assessment is complete.

### Step-by-Step Execution Flow:
1. **Artifact Ingestion:** `remediation_agent.py` initializes by parsing the existing `report.md` generated by the primary gatekeeper. It identifies the specific tables dropped/modified and the exact FQNs of the impacted downstream assets.
2. **Schema Extraction:** The agent interfaces with the OpenMetadata API to fetch the full physical schema definitions of the impacted downstream asset (and the original source table, if available).
3. **Local LLM Remediation Generation:** The agent constructs a strict system prompt containing the broken schema contract and sends it to a local **Ollama LLM** instance. The LLM is tasked with generating a backward-compatible SQL patch (e.g., a `CREATE VIEW` that mimics the dropped table's schema, mapping to the new structure) to guarantee zero downtime for downstream consumers.
4. **PR Feedback Loop:** The generated SQL patch is appended to the bottom of the existing `report.md` artifact. When the CI pipeline posts the PR comment, the developer receives both the block notification and the copy-pasteable SQL remediation.

## 3. Module Specifications: `remediation_agent.py`

### 3.1 Responsibilities
- Parse Markdown (`report.md`) to extract tabular impact data.
- Manage OpenMetadata API requests specifically for schema extraction.
- Orchestrate HTTP requests to the local Ollama LLM API.
- Perform string manipulation to append the remediation block to `report.md`.

### 3.2 Constraints
- **Strict Isolation:** Must operate entirely independently of the locked Phase 1-4 codebase. 
- **Security & Privacy:** Must strictly utilize a local Ollama LLM (e.g., `http://localhost:11434/api/generate`). No external or cloud LLM APIs (like OpenAI or Anthropic) are permitted, ensuring proprietary schema data remains entirely on-premises.
- **Fail-Safe Execution:** If the Ollama instance is unreachable, times out, or hallucinates invalid SQL, the agent must fail gracefully. It should append a generic warning to the report rather than crashing the CI pipeline, preserving the original Gatekeeper block.
